# DEV-TO-PLAN.md — "Make Ultimate Guitar Suck Less"

> Implementation plan for a Dev.to competition entry built with **GitHub Copilot CLI (free tier, Claude Haiku 4.5)**. Written for future-Haiku implementation: detailed by intent. If you're Haiku reading this — follow it linearly, escalate decisions tagged **[ESCALATE]**, and don't second-guess the architectural choices already made.

## TL;DR

A single Python CLI tool — `ug2songbook` — that takes a Tampermonkey-exported Ultimate Guitar bookmark JSON and emits **one standalone HTML file**. The HTML is a fully offline-capable songbook viewer with semantic search ("trist akustisk", "bryllupssang", "summer road trip"), chord diagrams, auto-scroll playback, and zero external dependencies. The user emails the HTML to themselves or their bandmates. That's the whole product.

The enrichment that powers the semantic search is generated by **looping copilot-cli with Claude Haiku** over each unique (artist, song) pair — a novel use of copilot-cli as a *batch LLM driver* rather than a code-suggestion tool.

This is also a **dogfooded prototype of NorTabs Phase 2.5** — the "Full Fat HTML export" already designed in `PLAN.md`. Shipping it as a contest entry first means we build the thing twice (once as a CLI, once as a proxy-backed web flow) and learn from the friction.

## Competition framing

**Contest theme**: *"Lag noe kult med copilot-cli gratis-versjonen"* (build something cool with the copilot-cli free tier).

**Why this fits the brief better than the typical entry**:
- Most entrants will use copilot-cli to *write their code*. Boring, predictable.
- We use copilot-cli as a **runtime tool** — a Haiku-powered enrichment workhorse the *end user's machine* invokes during the script run. That's a novel and demo-friendly angle.
- The deliverable is a **finished artifact a non-developer can use** (drop UG JSON in, get HTML out, email it to your band). Dev.to readers love things that ship.
- The "Make X suck less" frame is engagement-gold; UG's lock-in is a real, relatable pain.

**What we're NOT doing**:
- Not building a SaaS. The CLI tool runs locally, copilot-cli is on the user's machine, no servers, no signup.
- Not training a model.
- Not requiring an API key — copilot-cli free tier handles auth via the user's existing GitHub session.

## Why copilot-cli / Haiku for this

- **Free tier** — no API key, no per-token bill, no quota meter for the user.
- **Haiku is genuinely good enough** for English UG content. Already validated for Qwen3.6 at much lower quality in NorTabs (`PLAN.md` Phase 2.5 "Quality target"); Haiku beats Qwen3.6 on instruction-following + JSON discipline.
- **Already on developer machines** — most Dev.to readers either have copilot-cli installed or can install it in 30 seconds with `gh extension install github/gh-copilot`.
- **Latency-acceptable** for batch work — ~1-2 s per Haiku call, ~5-8 min total for a 250-tab personal library. Fits the "go grab a coffee" sweet spot.

**The novel angle for the article**: most uses of copilot-cli are interactive (`gh copilot suggest "kill process on port 8080"`). We're using it as a **prompt-runner subprocess** — `subprocess.run(["gh", "copilot", "suggest", "-t", "shell", PROMPT], capture_output=True)` or whichever invocation lets us pipe an arbitrary prompt in and get text out. **[ESCALATE]**: verify the exact copilot-cli invocation that accepts an arbitrary prompt and returns the model's response without shell-command framing. If `gh copilot` doesn't expose raw prompting, the fallback is `claude -p` via the Anthropic CLI — but the competition framing requires copilot-cli, so the first hour of implementation must confirm this works.

## Implementation phases

Do them in this order. Don't reorder — each phase verifies a piece that the next builds on.

### Phase 1 — copilot-cli prompt-runner spike (1-2 hours)

**Goal**: confirm copilot-cli can take an arbitrary prompt and return Haiku's text response.

1. Install: `gh extension install github/gh-copilot` if not already.
2. Try interactive: `gh copilot suggest "say hello"` — note the output framing.
3. Try non-interactive (script-friendly): test variations like:
   - `echo "PROMPT" | gh copilot suggest`
   - `gh copilot suggest -t shell "PROMPT"` (`-t shell` / `-t git` / `-t gh` are the documented types)
   - Any `gh copilot explain` variant that accepts free-form input
4. Pick the variant that returns Haiku's response with minimal framing chrome.
5. Write `crawler/copilot_cli.py` with one function:
   ```python
   def call_haiku(prompt: str, timeout: int = 60) -> str:
       """Run copilot-cli with `prompt`, return Haiku's raw text response."""
   ```
   - Use `subprocess.run(..., capture_output=True, text=True, timeout=timeout)`.
   - Strip CLI framing (banners, "⠋ Thinking..." spinners — copilot-cli writes these to stderr usually).
   - If copilot-cli returns non-zero, raise `RuntimeError` with stderr included.

**[ESCALATE]** if copilot-cli has no raw-prompt mode. Fallback options to discuss with the user: (a) use `claude -p --model haiku-4-5` via the Anthropic CLI (requires API key — kills the "no key needed" angle), (b) write the competition entry against a different free-tier CLI, (c) reframe as "use copilot-cli for codegen, use a different LLM for runtime enrichment" (weaker article angle).

### Phase 2 — enrichment driver (2-3 hours)

**Goal**: read UG JSON, loop over unique artists + songs, call `call_haiku()` for each, write per-entry results.

Create `crawler/ug2songbook.py`. Structure:

```
crawler/ug2songbook.py
├── main()
│   ├── parse args
│   ├── load UG JSON
│   ├── extract unique (artist, song) pairs + unique artist names
│   ├── enrich_artists()
│   ├── enrich_songs()
│   └── bundle_html()  # Phase 3
├── enrich_artists(artists, state, cli_fn)
├── enrich_songs(songs, bodies_by_id, state, cli_fn)
└── enrich_one(prompt, fallback, cli_fn)
```

**Heavy reuse from `crawler/enrich-private.py`** (in this repo) — do not reinvent:
- Copy `slugify()` verbatim.
- Copy `ARTIST_PROMPT` and `SONG_PROMPT` from `crawler/enrich.py` verbatim.
- Copy `extract_json()` from `crawler/enrich.py` (handles trailing prose + json5 fallback).
- Copy `salvage_truncated_json()` from `enrich-private.py` (handles Haiku running over token budget).
- Copy `STRICT_SUFFIX` from `enrich-private.py` — Haiku needs the same JSON discipline reminder.
- Copy `is_thin_fallback()` + `--retry-thin` flag plumbing.

**Differences from `enrich-private.py`**:
- Use `call_haiku()` from Phase 1 instead of `call_llm()` (which forks a subprocess differently).
- Write incremental state to a tempfile alongside the output HTML — survives interruption, resumable.
- Default to **serial calls** (not parallel) — copilot-cli's free tier almost certainly has concurrency limits or rate-limits. Add `--parallel N` flag for the brave.
- Pretty-print progress in the format shown in the UX walkthrough above (track + tab progress + per-call timing).

**CLI flags**:
- positional: path to UG JSON
- `--output PATH` (default: `ug-songbook-<YYYY-MM-DD>.html` in cwd)
- `--state PATH` (default: `<output>.state.json`, lets you resume after Ctrl+C)
- `--force` — re-enrich everything
- `--retry-thin` — only re-enrich entries that got the fallback (artist-name-only) result
- `--limit N` — stop after N enrichments (for testing)
- `--dry-run` — list what would be enriched, don't call LLM
- `--parallel N` (default 1)

### Phase 3 — HTML bundler (3-5 hours)

**Goal**: produce one self-contained `.html` file that opens to a working NorTabs viewer with the user's enriched private songbook pre-loaded.

This is the **"Full Fat HTML export"** already designed in `PLAN.md` Backlog under "Songbook HTML export — Lite shipped, Full Fat planned". Read that section before starting — it covers the high-level approach and the edge cases.

**Bundling strategy** (Python writes the HTML; no Node, no headless browser):

1. **Load `nortabs-web/index.html`** as the template.
2. **Walk the JS import graph** starting from `app.js`:
   - Recursive resolution of `import { ... } from './foo.js'` statements.
   - Topological sort so dependencies come before dependents.
   - Collect all `.js` files into a list.
3. **Concatenate** all `.js` modules into a single string, **stripping the `import`/`export` statements** (since they're meaningless inside a single inlined script). Replace `export function foo` with `function foo` and similar for `export const`, `export default`. **[ESCALATE]** the import-stripping if you hit re-exports or namespace imports (`import * as x`) — those need ad-hoc handling.
4. **Embed the catalog**: read `catalog.json` (~5 MB) and `enrichment.json` (if present), embed each as `<script type="application/json" id="embedded-catalog">…</script>`. Modify `catalog.js`'s `loadCatalog()` to check for the embedded block first.
5. **Embed the private bundle**: write the enriched private tabs as `<script type="application/json" id="embedded-private-bundle">…</script>`. Pre-populate `localStorage["nortabs:songbooks:v1"]` via an inline `<script>` so the UG songbook (`"ug-import-main"`) appears as a favorite on first open.
6. **Embed `style.css`** as `<style>…</style>`.
7. **Embed any SVG assets** (chord diagrams are generated in code; check whether `home-wordcloud.svg` is referenced — if so, inline as a data URL).
8. Write the resulting HTML to the output path.

**Reusable templates from existing code**:
- `exporter.js` (Lite version) already does a smaller version of this in JavaScript. Read it as reference for "what gets embedded" and "how localStorage prepopulation works". Do not try to call it from Python — port the relevant string templating to Python.
- The pre-populate-localStorage trick:
  ```html
  <script>
    if (!localStorage.getItem('nortabs:songbooks:v1')) {
      localStorage.setItem('nortabs:songbooks:v1', JSON.stringify({
        songbooks: [{ id: 'ug-import-main', name: 'Mine UG-importer', tab_ids: [...] }]
      }));
    }
    if (!localStorage.getItem('nortabs:private-tabs:v1')) {
      localStorage.setItem('nortabs:private-tabs:v1', JSON.stringify({
        version: 1,
        tabs: { ... }   // from the user's UG export, post-enrichment
      }));
    }
  </script>
  ```
- The search index in `search.js` already walks `nortabs:private-tabs:v1` at startup (per `PLAN.md` UG-import section) — so the moment the embedded private tabs land in localStorage, search picks them up. No extra wiring needed.

**Size budget**: 250 enriched tabs + catalog + viewer code ≈ 1.5-2.5 MB. Comfortably under Gmail's 25 MB attachment limit. If size becomes an issue, gzip the embedded JSON and inflate it client-side via `DecompressionStream` — see `PLAN.md` backlog "Take direct control of catalog/enrichment compression". Don't optimize prematurely.
